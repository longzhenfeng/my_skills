# 代码审查

把 review 的重点放在正确性、回归风险和缺失验证上，不把篇幅浪费在无关紧要的风格建议上。

## 审查目标

- 找出会导致错误结果、线上故障、兼容性问题或隐性回归的点。
- 找出测试覆盖不到的关键路径、边界条件和失败路径。
- 在证据不足时明确说明假设，不硬下结论。

## 默认工作方式

1. 先看改了什么，再判断这些改动可能影响哪些行为。
2. 优先检查正确性，再检查稳定性、性能、安全性和可维护性。
3. 多看新增逻辑、条件分支、状态流转、异常路径、数据结构变化和接口契约变化。
4. 如果有测试，核对测试是否真的覆盖关键行为，而不是只覆盖 happy path。
5. 如果缺少运行或验证条件，明确标记“未验证”。

## 重点检查项

### 正确性

- 条件判断是否遗漏空值、零值、边界值、类型变化。
- 新旧逻辑是否存在行为不一致。
- 重构是否改变了副作用顺序、返回值、默认值或错误处理。
- 并发、异步、重试、缓存、事务、幂等是否被破坏。

### 接口与兼容性

- API 参数、返回结构、字段语义是否发生不兼容变化。
- 数据库字段、配置项、环境变量、事件结构是否需要同步修改。
- 前后端、服务间、版本间是否存在契约漂移。

### 稳定性与安全

- 异常是否被吞掉，错误是否被错误降级。
- 权限、鉴权、输入校验、注入、泄漏敏感信息的风险是否增加。
- 资源释放、超时、重试、死循环、内存或连接泄漏是否存在风险。

### 测试

- 是否缺少关键分支、失败路径、边界条件的测试。
- 测试是否只验证表象，没有验证真正的业务结果。
- 快照类测试是否掩盖了真实行为变化。

## 输出规则

- findings 优先，按严重程度排序。
- 每条 finding 尽量包含：问题、影响、触发条件、依据。
- 能定位时带上文件和行号。
- 不要先写大段总结，不要先夸代码结构。
- 没发现问题时，要明确说“未发现明确缺陷”，并补充残余风险或未验证项。

## 输出模板

```markdown
1. [严重程度] 问题标题
   文件：path:line
   原因：这里写为什么是问题
   影响：这里写会导致什么后果
   建议：这里写最直接的修正方向
```

如果没有 findings，用这个格式：

```markdown
未发现明确缺陷。

残余风险：
- [未验证项或测试缺口]
```

## 审查倾向

- 默认把 bug、回归、数据错误、崩溃风险放在风格问题前面。
- 对“看起来能跑”的代码保持怀疑，重点看边界和失败路径。
- 不为了凑数量而报低价值问题。
- 如果某个风险需要运行、测试或更多上下文才能确认，要直接说清楚。

## 常见触发词

- “帮我 review 一下”
- “做个代码审查”
- “看看这个 PR 有没有问题”
- “帮我找 bug”
- “分析这次改动的风险”
